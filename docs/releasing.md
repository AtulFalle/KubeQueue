# Release process

KubeQueue publishes immutable artifacts from an exact `master` commit that has passed the
post-merge CI gate. Package manifest versions are private workspace metadata and remain `0.0.0`;
the release tag is the source of truth for container, Helm chart, and GitHub Release versions.

## Before starting

1. Merge all intended source changes through reviewed pull requests.
2. Wait for **KubeQueue CI** on the resulting `master` commit to succeed. This validates the exact
   commit, including PostgreSQL integration tests and generated-client consistency.
3. Confirm `CHANGELOG.md` describes the release. Changelog updates use the normal pull-request
   process and are not generated by release automation.
4. Confirm dependency and source security checks are clear.

## Start a release

Run the **Start KubeQueue Release** workflow and provide an unused semantic version tag with a `v`
prefix, such as `v0.2.0`.

The workflow resolves the current `master` SHA and refuses to continue unless that exact SHA has a
successful push CI run. It then dispatches **KubeQueue Release** with the immutable tag and SHA.
There is no release branch or release pull request.

## Publication safety

The publication workflow performs release-specific checks before creating the tag:

1. Revalidates that the selected SHA belongs to `master` and has successful push CI.
2. Verifies generated API client consistency and renders/packages the Helm chart with the requested
   version.
3. Builds each production image once under an immutable `sha-<commit>` tag, with provenance and an
   SBOM.
4. Scans all three images and smoke-tests the API and web images against PostgreSQL.
5. Creates the annotated release tag only after those checks pass.
6. Promotes the already-tested image manifests to the semantic version without rebuilding.
7. Publishes the versioned OCI chart and GitHub prerelease.

The workflow is rerunnable for the same tag and SHA. It rejects any attempt to point an existing tag
at different source.

## Version ownership

- Git tag `vX.Y.Z`: canonical product release identity.
- Container tags `X.Y.Z`: promoted from the tested `sha-<commit>` images.
- Helm chart `version` and `appVersion`: injected by the release workflow at package time.
- Workspace `package.json` versions: private development metadata fixed at `0.0.0`.
- OpenAPI `info.version`: API contract version, changed only when the contract version changes.

Chart image tags default to `.Chart.AppVersion`; operators can still override individual image tags
for custom builds.

## Verify

1. Confirm the GitHub release is marked as a prerelease and contains the chart and checksum.
2. Confirm the API, worker, web, and chart packages are associated with this repository and visible
   publicly. New GHCR packages may require a one-time visibility change in GitHub package settings.
3. Pull all three versioned images and verify their provenance/SBOM attestations.
4. Install `oci://ghcr.io/atulfalle/charts/kubequeue` into a clean namespace using a fresh
   PostgreSQL database and existing Kubernetes Secrets.
5. Port-forward the dashboard, submit one Job, confirm it reaches a terminal state, and uninstall
   the chart.
6. Verify the database and manually created Secrets remain after uninstall.

If publication fails after the tag is created, rerun **KubeQueue Release** with the same tag and SHA.
Never move or reuse an immutable release tag for changed source.
